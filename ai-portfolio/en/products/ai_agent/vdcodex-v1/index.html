DeepSeek Codex TUI

Terminal AI coding assistant, Python Textual + Rich TUI, 6 tools, 4x2 orthogonal mode system

DeepSeek Codex TUI v1.0 — Full Feature Report

1. Project Overview

DeepSeek Codex TUI is a terminal AI coding assistant similar to OpenAI Codex CLI, using the DeepSeek API as the backend (OpenAI protocol compatible). Core capability: AI reads/writes files, executes commands, and searches code through Tool Calls, interactively completing programming tasks in the terminal.

Tech Stack

ComponentTechnology
TUI frameworkPython Textual + Rich
HTTP clienthttpx (async streaming)
API protocolOpenAI compatible (DeepSeek / OpenAI / any compatible API)
Key storagekeyring (OS-level secure storage) + file fallback
SandboxDocker container (optional) / local subprocess

2. Core Features

2.1 Streaming Chat

  • Real-time streaming output of AI replies (Markdown rendered)
  • Thinking process (reasoning_content) collapsible display
  • Processing time display (“Sautéed for 3s”)
  • Token usage and cache hits updated in real time

2.2 Tool Calls

AI can call 6 tools to complete programming tasks:

3. 4x2 Orthogonal Mode System

3.1 Runtime Mode (What AI Is Doing)

ModeDescriptionToggle
autoAI executes tools normally per permission mode/plan to exit
planAI only plans, does not execute any tools/plan to enter

3.2 Permission Mode (How Tools Are Approved)

ModeFile ReadFile WriteCommand ExecutionSecurity Level
suggestConfirmConfirmConfirm5/5
auto-editAutoAutoConfirm3/5
full-autoAutoAutoAuto2/5
bypassAutoAutoAuto (dangerous)1/5

3.3 Approval Dialog (Claude Code Style)

● Write(calc.py)
  write_file(path='calc.py', content='...')
  Do you want to proceed?

  > 1. Yes
    2. Yes, and always allow for this session
    3. No
  • Yes — Allow for this instance
  • Yes, and always allow — Auto-approve same-type tools within the session
  • No — Reject, AI receives “User rejected this action”

8. Security Features

8.1 Docker Sandbox

  • Automatically uses container execution when Docker is available
  • python:3.12-slim image
  • --network none to disable networking
  • Working directory mounted to /workspace
  • Auto-cleanup of containers on exit
  • Falls back to local subprocess when unavailable

8.2 Dangerous Command Detection

DANGEROUS_COMMANDS = {
    "rm", "rmdir", "del", "rd", "format", "mkfs",
    "shutdown", "reboot", "halt", "poweroff",
    "sudo", "su", "runas",
    "chmod", "chown", "icacls",
    "kill", "killall", "taskkill",
}

8.3 API Key Secure Storage

  • Prefers OS keyring (Windows Credential Locker / macOS Keychain / Linux SecretService)
  • Falls back to encrypted file storage
  • Never hardcoded in source code

13. Known Limitations

  1. Docker dependency: Sandbox functionality requires Docker Desktop running
  2. No undo: Cannot automatically revert file writes
  3. No tests: Does not include automated tests
  4. Single user: Does not support concurrent multi-user access
  5. Offline unavailable: Requires network connection to API
