1. Project Overview

ALL-AGENT v5.0 is an AI Agent infrastructure platform, not a chatbot. It provides:

  • MCP tool calling — All external capabilities are exposed through standard MCP Servers
  • Dynamic tool loading — Only tools relevant to the current task are loaded, saving tokens
  • Coding + productivity workflows — Complete agent chains from planning to code to review
  • Security + permission boundaries — Three-tier permission model + Policy Engine
  • Swappable model providers — Seamless switching between Anthropic / OpenAI / Gemini / local models
  • Distributed DAG — Cross-machine task scheduling and execution
  • Self-optimizing runtime (Apotheosis) — Self-healing, self-evolving, self-adaptive

3.2 MCP Hub

Files: hub/client.py, hub/registry.py, hub/router.py, hub/gateway.py

Core router responsibilities:

  • Starts MCP Server processes (from config.yaml)
  • Auto-discovers tools via tools/list
  • Filters tools by task (determined by Orchestrator context layer)
  • Performs permission checks (Safe / Allow Edit / Bypass)
  • Evaluates Policy Engine rules
  • Routes tools/call to the correct Server
  • Handles Server crashes (auto-restart, max 2 times)
  • Enforces tool response truncation (hard limit per call)

Streaming Byte-Limit Guard:

  • 64KB chunked stdio reading
  • Exceeds 20MB physical limit -> closes pipe + kills MCP Server
  • Prevents OOM

Tool Truncation (Head-Tail + Semantic Anchoring):

  • When exceeding max_tokens, retains first 2K + last 8K tokens
  • Scans middle section for keywords like error/traceback/exception, extracts key lines and injects truncation markers
  • Eliminates “blind middle zone”

3.3 Runtime Modes

Files: runtime/hoshiai.py, runtime/swing.py, runtime/cadence.py, runtime/base_mode.py

ModeCore Mechanism
Hoshiai100% standalone operation, unlimited DAG, full agent chains
SwingMCP aggregation, Lease + Polling, PAUSED response wrapping, Base64 tunnel
CadenceDeadman Switch (30s lease / 45s no poll -> Abort), workspace isolation

Swing-specific:

  • Base64 tunnel: Auto-encodes code/text tool parameters in Base64, eliminating JSON/escaping character conflicts
  • Batch Tool Unpacker: Captures multiple tool intents in a single LLM turn, dispatches concurrently via anyio.create_task_group()

Cadence-specific:

  • Lease & Heartbeat: Each get_task_result poll renews lease for 30s, 45s without poll -> AbortEvent
  • PAUSED response wrapping: Includes instruction field, forces client LLM to stop generation and display the issue

4. Phase 0.5 — Layered Context Runtime

Files: orchestrator/context.py, orchestrator/context_snapshot.py

4.1 Four-Layer Context Hierarchy

L3: Long-term Knowledge
  └── Cross-session knowledge, user preferences, project structure
  └── Persistent storage, loaded on session start

L2: Task Memory
  └── Key information extracted per DAG task
  └── Agent: Planner loads L2+L3

L1: Rolling Summary
  └── Compressed session summary (old turns folded)
  └── Agent: Reviewer uses L1+L2

L0: Raw History
  └── Last N turns preserved in full
  └── Agent: Debugger uses L0+L1

4.2 Token Budget Control

Fixed overhead = system prompt + tool schemas
Available = total budget - fixed overhead - sunk_tokens
  ├── history: <= 60% of available (exceeding triggers compression)
  ├── output: reserve 20% of available
  └── buffer: reserve 20% (safety margin, never allocated)

4.3 Semantic Collapse

When history exceeds the 60% threshold:

  1. Lossless Metadata Condensation: Extracts incompressible entities before compression
    • All file paths and line ranges of successful modifications
    • All failed tool calls (name, arguments, error traceback)
    • Failure counter per task
  2. These metadata are appended as StructuredEvictionTombstones and never compressed
  3. Immutable Semantic Anchors: Architecture decisions, rejected alternatives, design constraints
    • Marked as PERMANENT, survive all compression cycles

4.4 Context Snapshot

Files: orchestrator/context_snapshot.py

  • ContextSnapshot: Saves complete L0-L3 state
  • create_snapshot(): Saves current context layers (including L1+L3)
  • restore_from_snapshot(): Restores to a specified snapshot
  • Supports Blackboard task memory injection

5. Phase 1 — Execution Runtime + Orchestrator

5.1 TaskGraph (DAG)

Files: orchestrator/task_graph.py

  • DAG definition + topological sort + parallel execution
  • TaskNode states: PENDING -> RUNNING -> COMPLETED / FAILED / SKIPPED
  • is_terminal(): Terminal state check
  • get_full_summary(): Full state summary (including retry_count, error, tokens_used, tools_called)
  • Fail-Fast Graph Pruning: FAILED nodes cascade all downstream to SKIPPED

5.2 Task Lifecycle

Files: orchestrator/task_lifecycle.py

Full state machine:

PENDING -> RUNNING -> WAITING_TOOL -> RUNNING -> COMPLETED
                       ↓                         ↓
                 BLOCKED_PERMISSION          PARTIAL
                       ↓                  WAITING_USER_INPUT
                  RETRYING -> RUNNING -> FAILED -> PAUSED

Resiliency Matrix:

StateTypeBehavior
CRASHEDTransientAuto-re-enqueue, exponential backoff (max 3 times)
PAUSEDTransientPreserves context snapshot, waits for explicit WakeEvent
WAITING_USER_INPUTTransient60s timeout -> Dummy Tool Responder
FAILEDTerminalImmediate block, cascades downstream to SKIPPED
PARTIALTerminalCannot re-enter, hard-links downstream to SKIPPED

5.4 Orchestrator

5.5 Recovery

Files: orchestrator/recovery.py

  • Checkpoint Atomicity: Copy-on-Write snapshot + fsync() + os.replace() atomic write
  • Sunk Token Registry: Tracks tokens consumed by crashed/retry/rejected operations
  • Sunk Cost Circuit Breaker: available / max_context_window < 0.35 -> stops retrying, marks PARTIAL
  • Recovery flow: Startup -> Load checkpoint -> Skip COMPLETED -> Re-enqueue FAILED/CRASHED

5.6 Blackboard

Files: orchestrator/blackboard.py

Three-layer shared data plane:

class Blackboard:
    task_state: dict[str, TaskNode]     # DAG node state
    memory: dict[str, MemoryEntry]      # Cross-agent discoveries, decisions
    artifacts: dict[str, Artifact]      # File references, code diffs, test output

Generation Ring Buffer: Each artifact retains latest + previous, supporting Diff/Regression/Blame.

GC Strategy:

  • task_state: Prune after all downstream nodes reach terminal state
  • memory: Auto-archive on same-key overwrite; when total exceeds threshold -> Semantic Merge
  • artifacts: After downstream references complete -> reclaim previous pointer

6.2 Agent Roles

AgentInputOutputFeatures
PlannerGoalDAG task graphCan run without provider (pure planning)
CoderTask + toolsCode changes + test resultsRequires provider, uses tools to edit files
SubReviewerCoder outputpass/rejectRequires provider, LLM review
GlobalReviewerAll SubReviewer resultsCross-task compatibilityNo provider defaults to pass
DebuggerFailure infoFix proposalRequires provider
FinalizerAll resultsFinal outputAggregates all artifacts

Agent Workflow (Initial DAG):

Planner -> DAG
          ├── Coder(task1) -> SubReviewer(task1) ──┐
          ├── Coder(task2) -> SubReviewer(task2) ──┤── GlobalReviewer -> Finalize
          └── Coder(task3) -> SubReviewer(task3) ──┘

Dynamic Node Expansion (Unrolling):

  • SubReviewer rejects -> Coder(v2) -> SubReviewer(v2)
  • GlobalReviewer rejects -> conflict_boundary -> only affected branches re-execute
  • DAG always remains acyclic; cycles are unrolled into sequential node pairs
