Astrognosy v2 - Feature Report

Project Overview

Astrognosy is a multi-model deep research tool similar to OpenAI Deep Research. When the user inputs a research question, the system automatically performs multiple rounds of searching, reading, and analysis, ultimately generating a cited, Markdown-formatted deep research report.

Key Differentiator: Supports all model providers compatible with the OpenAI API format (OpenAI, DeepSeek, Moonshot, Qwen, local Ollama, etc.) out of the box, with no lock-in to any specific platform.

Core Features

1. Research Loop Engine (Research Agent Loop)

The system automatically executes the following loop (up to 3-5 iterations):

User inputs question
    ↓
[Planning] LLM decomposes question into 3-5 sub-questions + search terms + search categories
    ↓
[Search] Parallel multi-source search: general + academic + GitHub + Wikipedia
    ↓
[Content Extraction] Three-tier fallback strategy to fetch web page content
    ↓
[Reading] LLM analyzes search results, extracts key facts and citations
    ↓
[Analysis] Evaluates information completeness, determines whether to continue searching
    ↓
  Sufficient information? --No--> Return to [Search]
    ↓ Yes
[Writing] Generates Markdown report with inline citations

Each step is pushed to the frontend in real time via SSE, allowing users to observe the complete research process.

Search Strategy:

  • Default multi-source concurrent: Searches all available categories simultaneously, results deduplicated by URL and aggregated
  • LLM intelligent routing: The planning-stage LLM decides which categories to search based on question type
  • Fallback mechanism: Automatically tries other providers in the same category when a provider fails

4. Designated Website Search (Domain Filtering)

Users can add a list of target domains in the settings panel (e.g., arxiv.org, github.com, nature.com), which are automatically applied during search:

  • DuckDuckGo / Brave / SearXNG: Uses site:xxx.com prefix
  • Tavily: Uses built-in includeDomains parameter
  • GitHub: Uses org: filtering
  • Providers that don’t support domain filtering (Wikipedia, OpenAlex) search normally

5. Browser Screenshots + Multimodal Analysis

Three-tier fallback content fetching strategy for handling JS-heavy rendered pages:

1. Jina Reader text extraction (fast, low cost, suitable for static pages)
   ↓ Insufficient content or extraction failed?
2. Playwright screenshot -> base64 -> multimodal LLM analysis (accurate, handles dynamic pages)
   ↓ Model doesn't support vision?
3. Playwright DOM text extraction (extracts content after removing scripts/styles/ads)

Multimodal Model Auto-Detection:

The system automatically detects whether the current model supports Vision capabilities. Known supported models:

VendorModel Keywords Supporting Vision
OpenAIgpt-4o, gpt-4-turbo, gpt-4-vision
Anthropicclaude-3, claude-3.5, claude-4
Googlegemini-1.5, gemini-2, gemini-pro-vision
Alibabaqwen-vl, qwen2-vl
Othersdeepseek-vl, llava, minicpm-v
  • Supports vision -> Screenshot approach available, green indicator shown in settings panel
  • Doesn’t support vision -> Text extraction only, gray indicator shown in settings panel
  • Users can manually toggle screenshot functionality in settings

6. Real-Time Progress Display

Frontend receives research progress via SSE, displaying:

  • Phase indicators: Planning -> Search -> Reading -> Analysis -> Writing -> Complete
  • Step-by-step details: Search terms, result counts, extraction status
  • Thinking process: Model analysis and judgment (if the model supports it)
  • Source list: All collected citation sources, color-coded by category
  • Timestamps: Precise time for each step

7. Research Report Rendering

Uses react-markdown + remark-gfm to render the final report:

  • Full Markdown support (headings, lists, tables, code blocks, blockquotes)
  • Dark theme adaptation
  • Inline citation links (click to jump to original source)
  • Citation source list (numbered to match inline citations)

Research Engine Prompt Design

Planning Stage (PLANNING_PROMPT)

Decomposes the user question into 3-5 sub-questions, each with search terms and search category (web/academic/code/knowledge), output in JSON format.

Reading Stage (READING_PROMPT)

Extracts key facts, citation sources, information credibility, and information gaps from search results, output in JSON format.

Analysis Stage (ANALYSIS_PROMPT)

Evaluates completeness of collected information (0-100 score), determines whether to continue searching, and if needed generates suggestions for the next round of search.

Writing Stage (WRITING_PROMPT)

Writes a structured Markdown research report based on all collected information, including inline citations [number] and a reference source list.

npm install

# 2. Install Playwright browser (only needed for screenshot functionality)
npx playwright install chromium

# 3. Configure environment variables (optional)
cp .env.example .env.local
# Edit .env.local and fill in your API Keys

# 4. Start the development server
npm run dev

# 5. Open in browser
# http://localhost:3000

Known Limitations

LimitationDescription
DuckDuckGo anti-scrapingFrequent requests may be rate-limited; consider configuring Tavily or Brave as alternatives
Screenshot performancePlaywright has some overhead launching Chromium; recommended to enable only when the model supports Vision
In-memory storageResearch reports are stored in memory and lost on page refresh (database could be added later)
SSE timeoutLong-running research (>5 minutes) may trigger proxy timeouts; timeout configuration needed
arXiv rate limitRequest interval of 3+ seconds recommended to avoid being blocked
