1. 项目总览

ALL-AGENT v5.0 是一个 AI Agent 基础设施平台，不是聊天机器人。它提供：

  • MCP 工具调用 — 所有外部能力通过标准 MCP Server 暴露
  • 动态工具加载 — 只加载当前任务相关的工具，节省 token
  • 编码 + 生产力工作流 — 从计划到代码到审查的完整 agent 链
  • 安全 + 权限边界 — 三级权限模式 + Policy Engine
  • 模型提供商可替换 — Anthropic / OpenAI / Gemini / 本地模型无缝切换
  • 分布式 DAG — 跨机器任务调度与执行
  • 自优化运行时（Apotheosis） — 自愈、自进化、自适应

TypeScript/TSX 前端20+

2. 核心架构：三轴 18 配置

系统由三个完全独立的轴控制，任意组合均有效：

轴选项范围
A: Security ModeSafe / Allow Edit / Bypass权限执行级别
B: Runtime ModeHoshiai / Swing / Cadence交付形式、状态机深度
C: Model StrategyAuto / Manual模型发现、配置、路由

A × B × C = 3 × 3 × 2 = 18 种运行配置。

轴 B 详解：Runtime 模式

模式定位DAG 深度Agent 数量压缩阈值
Hoshiai全能独立运行无限全部标准
SwingMCP 聚合适配器2 层受限标准
Cadence轻量独立运行受限减少40%

轴 C 详解：Model 策略

策略行为
Auto自动调用 /v1/models 发现模型，正则猜测能力，自动路由任务
Manual用户在 user_models.yaml 手动配置，零日兼容 + 安全回退

3.2 MCP Hub

文件: hub/client.py, hub/registry.py, hub/router.py, hub/gateway.py

核心路由器，职责：

  • 启动 MCP Server 进程（从 config.yaml）
  • 通过 tools/list 自动发现工具
  • 按任务过滤工具（Orchestrator context 层决定）
  • 执行权限检查（Safe / Allow Edit / Bypass）
  • 评估 Policy Engine 规则
  • 路由 tools/call 到正确的 Server
  • 处理 Server 崩溃（自动重启，最多 2 次）
  • 强制工具响应截断（硬限制 per call）

Streaming Byte-Limit Guard:

  • 64KB 分块读取 stdio
  • 超过 20MB 物理限制 → 关闭管道 + 杀死 MCP Server
  • 防止 OOM

Tool Truncation (Head-Tail + Semantic Anchoring):

  • 超过 max_tokens 时保留前 2K + 后 8K tokens
  • 中间扫描 error/traceback/exception 等关键词，提取关键行注入截断标记
  • 消除”中间盲区”

3.3 Runtime Modes

文件: runtime/hoshiai.py, runtime/swing.py, runtime/cadence.py, runtime/base_mode.py

模式核心机制
Hoshiai100% 独立运行，无限 DAG，完整 agent 链
SwingMCP 聚合，Lease + Polling，PAUSED 响应包装，Base64 tunnel
CadenceDeadman Switch（30s lease / 45s 无 poll → Abort），工作区隔离

Swing 特有:

  • Base64 tunnel: 代码/文本工具参数自动 Base64 编码，消除 JSON/转义字符冲突
  • Batch Tool Unpacker: 单次 LLM 轮次捕获多个工具意图，anyio.create_task_group() 并发分发

Cadence 特有:

  • Lease & Heartbeat: 每次 get_task_result 轮询续租 30s，45s 无轮询 → AbortEvent
  • PAUSED 响应包装: 包含 instruction 字段，强制客户端 LLM 停止生成并展示问题

4. Phase 0.5 — 分层上下文运行时

文件: orchestrator/context.py, orchestrator/context_snapshot.py

4.1 四层 Context 层次

L3: Long-term Knowledge
  └── 跨会话知识、用户偏好、项目结构
  └── 持久化存储，会话启动时加载

L2: Task Memory
  └── 每个 DAG 任务提取的关键信息
  └── Agent: Planner 加载 L2+L3

L1: Rolling Summary
  └── 压缩会话摘要（旧轮次折叠）
  └── Agent: Reviewer 使用 L1+L2

L0: Raw History
  └── 最近 N 轮完整保留
  └── Agent: Debugger 使用 L0+L1

4.2 Token 预算控制

固定开销 = system prompt + tool schemas
可用 = 总预算 − 固定开销 − sunk_tokens
  ├── history: ≤ 60% of available（超过 → 触发压缩）
  ├── output: 保留 20% of available
  └── buffer: 保留 20%（安全边际，永不分配）

4.3 Semantic Collapse

当 history 超过 60% 阈值：

  1. Lossless Metadata Condensation: 压缩前提取不可压缩实体
    • 所有成功修改的文件路径和行范围
    • 所有失败的工具调用（名称、参数、错误回溯）
    • 每个任务的失败计数器
  2. 这些元数据附加为 StructuredEvictionTombstones，永不压缩
  3. Immutable Semantic Anchors: 架构决策、被拒绝的方案、设计约束
    • 标记为 PERMANENT，在所有压缩周期中存活

4.4 Context Snapshot

文件: orchestrator/context_snapshot.py

  • ContextSnapshot: 保存 L0-L3 完整状态
  • create_snapshot(): 保存当前 context layers（含 L1+L3）
  • restore_from_snapshot(): 恢复到指定 snapshot
  • 支持 Blackboard 任务记忆注入

5. Phase 1 — 执行运行时 + 编排器

5.1 TaskGraph (DAG)

文件: orchestrator/task_graph.py

  • DAG 定义 + 拓扑排序 + 并行执行
  • TaskNode 状态：PENDING → RUNNING → COMPLETED / FAILED / SKIPPED
  • is_terminal(): 终态判断
  • get_full_summary(): 完整状态摘要（含 retry_count, error, tokens_used, tools_called）
  • Fail-Fast Graph Pruning: FAILED 节点级联所有下游为 SKIPPED

5.2 Task Lifecycle

文件: orchestrator/task_lifecycle.py

完整状态机：

PENDING → RUNNING → WAITING_TOOL → RUNNING → COMPLETED
                       ↓                         ↓
                 BLOCKED_PERMISSION          PARTIAL
                       ↓                  WAITING_USER_INPUT
                  RETRYING → RUNNING → FAILED → PAUSED

Resiliency Matrix:

状态类型行为
CRASHED瞬态自动重入队列，指数退避（最多 3 次）
PAUSED瞬态保留上下文快照，等待显式 WakeEvent
WAITING_USER_INPUT瞬态60s 超时 → Dummy Tool Responder
FAILED终态立即阻断，级联下游 SKIPPED
PARTIAL终态不可重入，硬链接下游 SKIPPED

5.4 Orchestrator

文件: orchestrator/orchestrator.py

中央调度器：

  • 接收请求 → DAG → 执行 → 结果
  • AgentResult: summary + status + artifacts + files_changed
  • ContextVar 传播到后台任务（copy_context().run()）
  • Background agent: 异步运行，结果写入 Blackboard

5.5 Recovery

文件: orchestrator/recovery.py

  • Checkpoint Atomicity: Copy-on-Write 快照 + fsync() + os.replace() 原子写入
  • Sunk Token Registry: 追踪 crashed/retry/rejected 消耗的 token
  • Sunk Cost Circuit Breaker: available / max_context_window < 0.35 → 停止重试，标记 PARTIAL
  • 恢复流程：启动 → 加载 checkpoint → COMPLETED 跳过 → FAILED/CRASHED 重入队列

5.6 Blackboard

文件: orchestrator/blackboard.py

三层共享数据平面：

class Blackboard:
    task_state: dict[str, TaskNode]     # DAG 节点状态
    memory: dict[str, MemoryEntry]      # 跨 agent 发现、决策
    artifacts: dict[str, Artifact]      # 文件引用、代码 diff、测试输出

Generation Ring Buffer: 每个 artifact 保留 latest + previous，支持 Diff/Regression/Blame。

GC 策略:

  • task_state: 所有下游节点达到终态后裁剪
  • memory: 同 key 覆盖自动归档，总量超阈值 → Semantic Merge
  • artifacts: 下游引用完 → 回收 previous 指针

6.2 Agent 角色

Agent输入输出特性
Planner目标DAG 任务图无 provider 也可运行（纯规划）
Coder任务 + 工具代码变更 + 测试结果需要 provider，使用工具编辑文件
SubReviewerCoder 输出pass/reject需要 provider，LLM 评审
GlobalReviewer所有 SubReviewer 结果跨任务兼容性无 provider 默认 pass
Debugger失败信息修复方案需要 provider
Finalizer所有结果最终输出汇总所有 artifacts

Agent 工作流（初始 DAG）:

Planner → DAG
          ├── Coder(task1) → SubReviewer(task1) ──┐
          ├── Coder(task2) → SubReviewer(task2) ──┤── GlobalReviewer → Finalize
          └── Coder(task3) → SubReviewer(task3) ──┘

动态节点展开（Unrolling）:

  • SubReviewer 拒绝 → Coder(v2) → SubReviewer(v2)
  • GlobalReviewer 拒绝 → conflict_boundary → 仅受影响分支重执行
  • DAG 始终保持无环；周期展开为顺序节点对

13. 已知局限与后续方向

已知局限

局限严重度说明
Blackboard GC 需更大重构中task_state 保留策略需按 DAG 拓扑动态调整
Context snapshot 不完整低恢复时 L1+L3 已支持，但压缩比可优化
Router 无 per-agent filter 实例低当前使用无状态函数调用，多次过滤无缓存
context_usage 为轮询而非推送低前端每 5s 轮询，可改为 SSE
